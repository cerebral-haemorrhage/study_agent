# Agent 开发教学笔记

> 基于 `handcraft_agent`（手搓版）与 `langgraph_agent`（框架版）的对比学习

---

## 一、两版总览：手搓版 vs 框架版

> 一句话总结：手搓版 = 你自己写 for 循环控制 LLM 来回思考；框架版 = 你画一张"流程图"，LangGraph 帮你跑。

### 整体对应关系

| 手搓版 | 框架版 |
|--------|--------|
| `react_loop()` 整个函数 | 一次 `graph.invoke()` 内部: agent → tools → agent → ... → `__end__` |
| `for turn in range(MAX_TURNS)` | 图的循环边 `tools → agent`（`add_edge`） |
| `react_memory` 字符串 | `state["messages"]`（由 LangGraph 自动维护） |
| `ConversationMemory` 列表 | `MemorySaver` + `thread_id`（跨多次 invoke 的记忆） |
| 手动拼消息再发给 LLM | LangGraph 内部自动从 state 拿消息发给 LLM |

## 二、工具定义：@tool 装饰器 vs 手写文本

| 手搓版 | 框架版 |
|--------|--------|
| 手写一段文本 `TOOLS_DESC`，LLM 看到文字才知道有工具 | `@tool` 装饰器自动生成 JSON Schema，LLM 收到结构化数据 |
| 加新工具要改 3 处：`tools.py`（函数+描述）、`react.py`（`elif` 分支） | 加新工具只改 1 处：`tools.py`（写函数 + 加入 `TOOLS` 列表） |

```python
# 手搓版：手写工具描述文本
TOOLS_DESC = """
工具名: get_weather
描述: 查询指定城市的天气
参数: city (字符串)
"""

# 框架版：@tool 装饰器自动搞定
@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气。"""
    return WEATHER_DB.get(city, "...")
```

## 三、LLM 输出解析（两版最大区别）

| 手搓版 | 框架版 |
|--------|--------|
| LLM 输出纯文本字符串 | LLM 输出带 `tool_calls` 的结构化对象 |
| 自己 `parse_react_output()` 逐行匹配解析 | 直接读 `msg.tool_calls[0]["name"]` |

```python
# 手搓版：LLM 返回一串字符串，要自己解析
"Thought: 查天气
Action: get_weather
Action Input: {"city": "北京"}"
→ 自己写 parse_react_output() 去拆这串文本

# 框架版：LLM 返回结构化对象
AIMessage(
    content="我来帮你查天气",
    tool_calls=[{"name": "get_weather", "args": {"city": "北京"}}]
)
→ 直接 msg.tool_calls[0]["args"] 就能拿到参数
```

**关键**：`llm.bind_tools(TOOLS)` 这一行，让 LLM 不再输出自由文本，而是输出带工具调用的结构化消息。

## 四、StateGraph 三板斧：State、Node、Edge

| 概念 | 解释 | 代码对应 |
|------|------|----------|
| **State** | 整张图流转的"记忆"，每个节点都能读写 | `AgentState` |
| **Node** | 一个处理步骤，接收 State 返回 State | `agent_node`, `tools_node` |
| **Edge** | 节点之间的路线，"做完A下一步去哪" | `add_edge`, `add_conditional_edges` |

```
    ┌─────────┐
    │  agent   │ ← 思考节点
    └────┬────┘
         │ should_continue 判断
    ┌────┴────┐
    │         │
  要工具    不要
    │         │
    ▼         ▼
  tools      END
    │
    └──→ 回到 agent（形成循环！这就是 ReAct）
```

**省了什么活？** 不用写 `for turn` 循环了，在"画蓝图"而不是"写流程"。LangGraph 按蓝图自动跑到 `__end__` 为止。

## 七、ReAct 三件套：Thought → Action → Observation

ReAct 模式的最小工作单元：

| 环节 | 做的事 | 谁执行 |
|------|--------|--------|
| **Thought** | "现在情况是什么？下一步做什么？" | LLM |
| **Action** | "具体调哪个工具？参数是啥？" | LLM 决定，系统执行 |
| **Observation** | "工具返回了什么？" | 系统执行后填回去 |

LLM 本身不知道实时数据，整个链条就是：

> LLM 说"我需要查一下" → 系统帮它查 → 把结果贴回给 LLM → LLM 基于结果回答

一次 invoke 里这三步可以循环多次。框架版对应 `debug_trace.py` 里的 while 循环：`agent_node` → `should_continue` → `tools_node` → 回到 `agent_node`...

## 八、多轮对话与消息 ID 问题

### 两种"记忆"

| | ReAct 记忆 | 对话记忆 |
|---|---|---|
| **作用域** | 单次请求内 | 跨多轮对话 |
| **手搓版变量** | `react_memory`（字符串） | `ConversationMemory.messages`（列表） |
| **框架版对应** | 一次 invoke 内部自动循环 | 多次 invoke 之间需要 MemorySaver |

### 消息 ID 对不上的原因

DeepSeek/OpenAI API 要求 tool 回复必须绑定到对应的 tool_call 请求，靠 `id` 配对：

```
正常:
  AIMessage(id="call_abc123", tool_calls=[{"id":"call_abc123"}])
  ToolMessage(tool_call_id="call_abc123")
  → id 对上，没问题 ✓

报错场景:
  第一次 invoke: LLM 返回 AIMessage(id="call_abc123")
                 Tools_node 返回 ToolMessage(tool_call_id="call_abc123")
  第二次 invoke: main.py 把旧的 AIMessage + ToolMessage 又塞进 messages
                 agent_node 全部发给 DeepSeek
                 DeepSeek: "这次请求里没找到 id='call_abc123' 的 AIMessage！"
                 → 报错 ❌
```

**本质**：`AIMessage(tool_calls)` 和 `ToolMessage` 是配对出现的，ID 只在同一次 API 请求内有效，不能跨请求复用。

### 解决方案：MemorySaver

```
当前做法:
  第一轮: main.py → graph.invoke(全部历史)
  第二轮: main.py 手动攒历史 → graph.invoke(全部历史) ❌ id 对不上

MemorySaver 做法:
  第一轮: main.py → graph.invoke({messages: [新问题]}, thread_id="t1")
          跑完后 MemorySaver 存 state(key="t1")
  第二轮: main.py → graph.invoke({messages: [新问题]}, thread_id="t1")
          只传新输入，不传历史！
          LangGraph 内部通过 thread_id 找到旧 state，追加新消息后"续跑"
          因为续跑而非全新 run，id 全部有效 ✓
```

**类比**：
- 当前做法 = 每次打电话都重新说明全部前情提要，夹带了上通电话的内部编号
- MemorySaver = 接线员记录了上一通电话，第二通不需要重复说

## 九、debug_trace.py：调试包装器

框架版 `graph.invoke()` 默认是黑盒——只看到最终结果。`debug_trace.py` 用**包装器模式**手动模拟图的执行流程，每一步打印，和手搓版 `verbose=True` 效果一样。

```python
class DebugWrapper:
    def invoke(self, state, config=None):
        while True:
            result = agent_node(current_state)      # LLM思考 → 打印
            decision = should_continue(state)       # 判断 → 打印
            if decision == "__end__": break
            result = tools_node(state)              # 执行工具 → 打印
            # ← 循环
```

```python

react_memory += f"\n{llm_output}"           # 把 Thought 记下来

if parsed["action"]:                        # 如果 LLM 说要调工具
    obs = execute_tool(parsed["action"], ...)  # 真的去执行 → Action

```

──────────────────────────────────
🔄 Round 1/10
──────────────────────────────────
📝 LLM 输出:
  Thought: 用户想知道当前时间，我需要使用 get_time 工具。
  Action: get_time
  Action Input: {}

🔧 执行工具 [get_time]: {}
📊 Observation: 2026-05-23 00:00

三件套拆开看：

| 环节 | 问 LLM 的问题 | LLM 的回答 | 谁执行 |
|------|-------------|-----------|--------|
| **Thought** | "现在情况是什么？下一步做什么？" | "用户要时间，我该调 get_time" | LLM |
| **Action** | "具体调哪个工具？" | "get_time，没参数" | LLM *决定*，系统 *执行* |
| **Observation** | "工具返回了什么？" | "2026-05-23 00:00" | 系统（实际执行后填回去） |

     ToolMessage(tool_call_id="call_abc123", ...)]      ← 系统说"这是结果"

    DeepSeek 检查: ToolMessage 的 tool_call_id = "call_abc123"
                  前面确实有一个 AIMessage 写了同样的 id

    ToolMessage(tool_call_id="call_abc123", ...),       ← 旧工具结果
    HumanMessage("那明天天气呢"),                        ← 新用户输入

graph.invoke({"messages": messages})  ← 全新的一个 run

# DeepSeek 看到:
#   ToolMessage(tool_call_id="call_abc123") 
#   但是在它收到的这段对话中，没有找到 id = "call_abc123" 的 AIMessage
#   因为"call_abc123"是上一轮 graph run 里 LLM 生成的，
#   跟这次新的 run 完全无关

> 你和朋友轮流说话，每个人说的每句话带一个编号。

**本质：** `AIMessage(tool_calls)` 和 `ToolMessage` 是**配对出现**的，ID 只在**同一次 API 请求内**有效，不能跨请求复用。

## 问题 2：MemorySaver 替你记是什么意思？

### 当前做法 vs MemorySaver 做法

**当前做法（main.py 手工攒历史）：**

```
第一轮:
  main.py ⟹ graph.invoke({"messages": [HumanMessage("现在时间")]})
  graph 跑完 → 返回最终 state
  main.py 把 state["messages"] 存到变量 messages 里

第二轮:
  main.py 把 messages + 新输入 拼好再传回去
  main.py ⟹ graph.invoke({"messages": 旧历史 + 新输入})
  graph 收到了旧历史 → 里面夹着上一轮的 AIMessage+ToolMessage → ❌ 报错
```

**MemorySaver 做法：**

```
第一轮:
  main.py ⟹ graph.invoke(
      {"messages": [HumanMessage("现在时间")]},
      {"configurable": {"thread_id": "t1"}}
  )
  graph 跑完后 → MemorySaver 自动把最终 state 存下来（key = "t1"）

第二轮:
  main.py 什么都不用管，只传新输入:
  graph.invoke(
      {"messages": [HumanMessage("那明天天气呢")]},  ← 只传新问题
      {"configurable": {"thread_id": "t1"}}          ← 同一个 thread_id
  )

  LangGraph 内部做的事:
    1. 通过 thread_id="t1" 去 MemorySaver 里查到: "哦，t1 之前的 state 是这样的"
    2. 取出旧的 state，把新 HumanMessage 追加到末尾
    3. 从 agent_node 开始"续跑" → 不是全新 run，是同一 run 的延续
  
  那旧 tool_calls 的 ID 呢？
    因为是续跑，state 里的 AIMessage(id="call_abc123") 和
    ToolMessage(tool_call_id="call_abc123") 仍然是有效的，
    所以 LLM 不会报错 ✓
```

**类比：**

> 当前做法 = 你每次打电话（graph run）都重新说明全部前情提要，结果夹带了上通电话的编码，对方听不懂
>
> MemorySaver = 接线员（LangGraph）帮你记录了上一通电话的通话记录（state）
> 你第二通电话不需要重复说，接线员说"我查一下记录 → 哦他上次说到了这里 → 让他继续说"
